# Releasing LoRa Ranch Sentinel

A "release" here is one GitHub release: a tagged git commit + a set of
flashable `.img` files + a `SHA256SUMS` file, all attached to a public
download page. Operators flash from a release; developers reference a
tag when reporting issues.

This doc is the end-to-end checklist. The mechanical bits are in
[`scripts/cut_release.sh`](../scripts/cut_release.sh) — the doc covers
what the script doesn't (deciding on a version number, writing
release notes, verifying after publish).

## Versioning

SemVer with a `v` prefix: `v0.1.0`, `v1.0.0`, `v1.2.3`. Prereleases
are allowed as suffixes: `v1.0.0-rc1`, `v1.0.0-beta`.

Rough rules of thumb for this project:

- **Patch** (`v1.0.0` → `v1.0.1`) — bug fixes, doc tweaks, internal
  refactors. Operators flashing patch updates should see no surprises.
- **Minor** (`v1.0.0` → `v1.1.0`) — new features (new Telegram
  command, new sensor type, new build profile). Existing deployments
  keep working.
- **Major** (`v1.0.0` → `v2.0.0`) — breaking changes. Config-file
  format changes, on-disk schema changes that aren't auto-migrated,
  hardware compatibility changes. Document the migration path in the
  release notes.

Hobby project, you're the only operator. Don't agonize over the
distinction — the badge just needs *something* sortable.

## What goes in a release

Default: production images only.

```
releases/
  base_station_pi3.img        — Pi 3 / 3B+ base station
  gate_client_pi0w.img         — Pi Zero W / Zero 2 W gate monitor
  SHA256SUMS                   — generated by cut_release.sh
```

Dev images (`*_dev.img`) are **held back by default**. They ship with
a known root password (`root:ranchdev`), an SSH server, and the
dev-diag overlay — none of which belong on a public download page.
If you really need them (for example, to repro a hardware bug a
collaborator hit), pass `--include-dev` to `cut_release.sh` and call
that out in the release notes.

## Cutting a release — step by step

### 1. Build the images for the tag you want to release

```bash
RANCH_BUILD_TARGETS="base gate" \
RANCH_BUILD_PROFILE=production \
RANCH_BUILD_HOST=192.168.1.249 \
RANCH_BUILD_USER=vijit \
./remote_build.sh
```

This drops `base_station_pi3.img` and `gate_client_pi0w.img` into
`releases/`. Full build options (profiles, single-target builds,
the remote-build pre/post hooks, passwordless-sudo setup) live in
[docs/BUILDING.md](BUILDING.md).

### 2. Make sure your working tree is clean

```bash
git status         # should be empty
git log -1         # confirms which commit the tag will pin to
```

If you have uncommitted changes, commit them first. The release tag
pins to your current `HEAD`; uncommitted changes wouldn't be captured
by the tag, so anyone who later cloned at that tag would see a
different repo than what produced the image.

### 3. Run the cut-release script

```bash
./scripts/cut_release.sh v0.1.0
```

The script does:

1. **Pre-flight checks.** `gh` installed and authenticated; clean
   working tree; tag doesn't already exist locally or on origin;
   `releases/` has at least one production image.
2. **Image verification.** Runs `scripts/verify_image.sh` against
   each `.img` it's about to publish (`--skip-verify` to bypass).
3. **SHA256SUMS.** Writes `releases/SHA256SUMS` covering every asset
   that's about to be uploaded.
4. **Summary + confirmation.** Prints the planned tag, commit,
   assets, and a y/N prompt. Nothing destructive has happened yet.
5. **Tag + push tag.** `git tag -a` annotated tag, `git push origin
   refs/tags/v0.1.0`.
6. **`gh release create --draft`.** Uploads every asset to the
   release. The release is a **draft** — visible only to repo
   collaborators on github.com, hidden from the public until you
   click Publish.

The script prints the URL of the draft release at the end.

#### Useful flags

- `--dry-run` — show the plan without tagging or uploading. Good
  for a final sanity check.
- `--notes-file CHANGELOG.md` — use a markdown file as the release
  body (instead of auto-generating from commits).
- `--include-dev` — also upload the `*_dev.img` files. Default
  behavior is to skip them.
- `--allow-dirty` — proceed even with uncommitted changes. Avoid
  unless you have a specific reason.
- `--skip-verify` — skip the `verify_image.sh` step (e.g. if you
  already ran it manually). Verify failures are rare but the check
  is cheap and would catch e.g. a dev artifact in a prod image —
  don't skip routinely.

### 4. Review the draft release on github.com

Open the URL the script printed. You'll see:

- The tag name and the commit it points to.
- Auto-generated release notes from commits since the last tag
  (or your `--notes-file` content).
- The image and SHA256SUMS assets attached.

Edit the release notes in the GitHub web UI. Useful sections to
add by hand:

- **Highlights** — 1–3 bullets for what an operator should know.
- **Verified hardware** — what you actually tested this release on.
- **Migration notes** — if any operator action is required to
  upgrade (e.g. captive-portal re-entry, factory reset, re-pair).
- **Known limitations** — what's still pending real-hardware
  validation. The Readme.md "What's verified today" section is
  the canonical public list.
- **Flashing instructions** — the
  `flash_base_station.py` / `provision_gate.py` command lines or a
  link to [docs/USER_GUIDE.md](USER_GUIDE.md).

### 5. Publish

Click **Publish release** in the GitHub web UI. The release becomes
public; download counts start ticking; the `Latest release` badge in
the README updates within a minute.

If you spot a problem after publishing, you can delete the release
(GitHub web UI → release → trash icon). You'll also want to delete
the tag if you plan to reuse the version number:

```bash
git push --delete origin v0.1.0
git tag -d v0.1.0
```

Deleting a published release after operators have started downloading
is bad form — they'll keep referencing a URL that 404s. Prefer a
quick `v0.1.1` patch over un-publishing `v0.1.0`.

## Operators: verifying a download

Anyone flashing your image can confirm what they downloaded matches
what you signed off on:

```bash
shasum -a 256 -c SHA256SUMS
```

Expected output:
```
base_station_pi3.img: OK
gate_client_pi0w.img: OK
```

The README's "I want to flash a pre-built image" section points
operators at the latest release page and this verify step.

## Release notes templates

### Minor release (new feature)

```markdown
## Highlights

- New `/foo` Telegram command — does X.
- Y now works on Z hardware.

## Verified on

- Base: Pi 3B+, 4GB SD
- Gate: Pi Zero 2 W, reed switch on BCM 21

## Flashing

Production images attached. See
[docs/USER_GUIDE.md](https://github.com/VijitSingh97/gate-checker/blob/v0.2.0/docs/USER_GUIDE.md)
for the captive-portal walkthrough.

## What changed

(auto-generated commit list will go here from --generate-notes)
```

### Patch release (bug fix only)

```markdown
## What's fixed

- `/status GATE-X` no longer double-pages when the gate is in the
  same state as last recorded (Session 16 dedup regression).

## Verified on

(same as v0.1.0)

## Upgrading

Re-flash the base station. No re-pairing needed; existing gate
Fernet keys and event history are preserved across re-flash IF you
use `flash_base_station.py` against your existing SD card with the
same per-device sticker.

## What changed

(commits)
```

### Major release

Document the migration path explicitly. If config files, on-disk
schemas, or gate-side firmware changed in a breaking way, spell out
exactly what operators have to do — factory reset, re-pair every
gate, etc. Use the "Reply" examples from
[docs/TELEGRAM.md](TELEGRAM.md) to make it copy-paste-able.

## Bumping the test-suite count

After a release, the README's quick claims (e.g. "159 stdlib-unittest
tests") will drift. Either keep them deliberately vague ("100+ tests")
or update them as part of the release prep. The authoritative live
count is whatever `scripts/run_tests.sh` reports at HEAD; check it
before editing the README and TELEGRAM.md to bump the number.
